1. 프로젝트 소개 — 이 프로그램은 뭘 하는 건가요?

이 프로그램은 자동차 튜닝 부품 판매점에서 사용하는 POS(Point of Sale) 시스템입니다. 편의점 계산대의 터치스크린 프로그램과 비슷하지만, 웹 브라우저에서 돌아간다는 것이 특별합니다.

4. React 핵심 개념 — 코드를 읽기 전에 알아야 할 것

React는 화면을 만드는 라이브러리입니다. 이 프로젝트를 이해하려면 4가지 핵심 개념만 알면 됩니다.

4-1. 컴포넌트 (Component) = 레고 블록

화면의 한 조각을 함수로 만든 것입니다. 하나의 .jsx 파일 = 하나의 컴포넌트입니다.

// Toast.jsx — 알림 메시지를 보여주는 컴포넌트
export default function Toast({ message, type }) {
  return (
    <div className={`toast toast-${type}`}>
      {message}
    </div>
  );
}
JSX = JavaScript + HTML 합친 것입니다. <div>처럼 HTML을 쓰지만, {변수}로 JavaScript 값을 끼워넣을 수 있습니다. 처음엔 이상하게 보여도 금방 익숙해집니다!

4-2. Props = 부모가 자식에게 전달하는 데이터

컴포넌트에 데이터를 넘기는 방법입니다. HTML의 속성(attribute)과 비슷합니다.

// App.jsx에서 Dashboard에 데이터를 넘김
<Dashboard
  orders={orders}           // 주문 목록
  products={products}       // 상품 목록
  setCurrentPage={setCurrentPage}  // 페이지 이동 함수
/>

// Dashboard.jsx에서 받아서 사용
export default function Dashboard({ orders, products, setCurrentPage }) {
  return <div>총 주문: {orders.length}건</div>
}
App.jsx (부모) │ ├── orders, products, customers ← "데이터 보관소" 역할 │ ├── 📦 props 전달 │ ├── Dashboard ← orders, products, setCurrentPage ├── MainPOS ← products, cart, saveOrder ├── OrderHistory ← orders, setOrders ├── CustomerList ← customers, setCustomers └── AdminPage ← products, customers, supabase

4-3. State (상태) = 변할 수 있는 데이터

useState로 만드는 변수입니다. 값이 바뀌면 화면이 자동으로 다시 그려집니다.

const [cart, setCart] = useState([]);
//  ↑ 현재 값  ↑ 값 변경 함수      ↑ 초기값 (빈 배열)

// 장바구니에 상품 추가
setCart([...cart, newItem]);   // → 화면이 자동으로 업데이트됨!
❌ 일반 변수
let count = 0;
count = 5;
→ 값은 바뀌지만 화면은 그대로
✅ State
const [count, setCount] = useState(0);
setCount(5);
→ 값이 바뀌면 화면도 자동 업데이트

4-4. useEffect = "이런 상황이 되면 이걸 해라"

컴포넌트가 화면에 나타났을 때, 또는 특정 값이 바뀌었을 때 자동으로 실행되는 코드입니다.

// App.jsx — 앱이 처음 시작될 때 Supabase에서 데이터 로드
useEffect(() => {
  async function loadData() {
    const prods = await supabase.getProducts();   // DB에서 상품 가져오기
    const ords  = await supabase.getOrders();     // DB에서 주문 가져오기
    const custs = await supabase.getCustomers();  // DB에서 거래처 가져오기
    if (prods) setProducts(prods);
    if (ords)  setOrders(ords);
    if (custs) setCustomers(custs);
  }
  loadData();
}, []);  // ← 빈 배열 = 처음 한 번만 실행
의존성 배열의미실행 시점
[] (빈 배열)"처음 한 번만"컴포넌트가 화면에 나타날 때
[count]"count가 바뀔 때마다"count 값이 변경될 때
생략"항상"화면이 다시 그려질 때마다 (주의!)

5-2. 라우팅 — URL 없이 페이지 전환하기

보통 웹사이트는 /about, /contact 같은 URL로 페이지를 나눕니다. 이 프로젝트는 더 간단한 방법을 씁니다.

const [currentPage, setCurrentPage] = useState('dashboard');

// 페이지 전환: setCurrentPage('pos') → POS 화면으로 이동

// 렌더링: currentPage 값에 따라 다른 컴포넌트를 보여줌
switch (currentPage) {
  case 'dashboard': return <Dashboard ... />;
  case 'pos':       return <MainPOS ... />;
  case 'orders':    return <OrderHistory ... />;
  // ... 나머지 페이지들
}
이 방식을 SPA (Single Page Application)라고 합니다. 한 페이지 안에서 내용만 바꿔치기하므로 페이지 이동이 빠르고 깜빡임이 없습니다.

5-3. 주문 저장 흐름 — saveOrder()

주문 버튼을 누르면 어떤 일이 벌어지는지 순서대로 살펴봅시다.

1
사용자가 장바구니에 상품을 담고 "주문 확인" 버튼을 누름
2
saveOrder() 함수가 호출됨
3
같은 고객의 오늘 날짜 주문이 이미 있는지 확인 → 있으면 병합 (아이템 합치기)
4
없으면 새 주문 생성 → Supabase DB에 저장
5
deductStock()으로 재고 자동 차감
6
Supabase 실시간 구독이 변경을 감지 → 다른 기기에서도 자동 반영
날짜 비교 시 getTodayKST() 함수를 사용합니다. JavaScript의 new Date()는 UTC 기준이라, 한국 시간 자정~오전 9시에 "오늘"이 "어제"로 계산되는 버그가 있었습니다. 이를 수정한 것이 KST 타임존 함수입니다.

6. 페이지별 기능 해설

6-1. MainPOS.jsx — POS 계산기 (1,074줄)

이 프로그램의 핵심 기능입니다. 상품을 선택하고, 수량을 정하고, 주문을 완성합니다.

화면 구성

┌──────────────────────────────────────────┐ │ 상단: 검색바 + 필터 │ ├────────────────────┬─────────────────────┤ │ │ │ │ 왼쪽: 상품 그리드 │ 오른쪽: 장바구니 │ │ (카테고리별 분류) │ (선택한 상품 목록) │ │ │ 수량 조절 │ │ 상품 카드 클릭 │ 가격 합계 │ │ → 장바구니에 추가 │ │ │ │ [주문 확인] 버튼 │ ├────────────────────┴─────────────────────┤ │ 하단: 주문 확인 모달 │ │ 거래처 선택 → 최종 확인 → 저장 │ └──────────────────────────────────────────┘

핵심 코드 패턴: 장바구니 추가

const addToCart = (product) => {
  // 이미 장바구니에 있으면 수량만 +1
  const existing = cart.find(item => item.id === product.id);
  if (existing) {
    setCart(cart.map(item =>
      item.id === product.id
        ? { ...item, quantity: item.quantity + 1 }  // 수량 증가
        : item                                       // 나머지는 그대로
    ));
  } else {
    // 없으면 새로 추가
    setCart([...cart, { ...product, quantity: 1 }]);
  }
};
{ ...item, quantity: item.quantity + 1 } 은 "item을 복사하되, quantity만 바꿔라"라는 뜻입니다. 이것을 스프레드 연산자라고 하며, React에서 가장 많이 쓰는 패턴입니다.

6-2. OrderHistory.jsx — 주문 내역 (839줄)

핵심 기능: 날짜 필터

// 필터링: 원본 데이터를 건드리지 않고, 보여줄 데이터만 걸러냄
const filteredOrders = useMemo(() => {
  return orders
    .filter(o => {  // 1. 날짜 필터
      if (dateFilter === 'today') return orderDate === today;
      if (dateFilter === 'week')  return isWithinWeek(orderDate);
      // ...
    })
    .filter(o => {  // 2. 검색어 필터
      return o.customer_name?.includes(searchQuery);
    })
    .filter(o => {  // 3. 반품 필터 (반품 카드 클릭 시)
      return !showReturnsOnly || (o.totalReturned > 0);
    });
}, [orders, dateFilter, searchQuery, showReturnsOnly]);
//          ↑ 이 값들이 바뀔 때만 다시 계산 (성능 최적화)
useMemo는 "결과를 기억해두고, 입력이 바뀔 때만 다시 계산해라"라는 뜻입니다. 주문이 수백 건일 때 매번 필터링하면 느려지므로, 필요할 때만 계산합니다.

카드 → 모달 패턴

// 카드 컴포넌트

7. Supabase 연동 — 데이터는 어디에 저장되나요?

Supabase는 무료 클라우드 데이터베이스입니다. 구글 스프레드시트처럼 표(테이블) 형태로 데이터를 저장하지만, 프로그램에서 자동으로 읽고 쓸 수 있습니다.

7-1. 테이블 구조

테이블역할주요 컬럼
products상품 목록id, name, wholesale(도매가), retail(소매가), stock(재고), category
orders주문 내역id, customer_name, items(JSON), total, created_at
customers거래처id, name, phone, address, memo
saved_carts저장 장바구니id, name, items(JSON), created_at

7-2. CRUD 함수 (supabase.js)

CRUD = Create(생성), Read(읽기), Update(수정), Delete(삭제)의 약자입니다.

// supabase.js — REST API를 감싼 함수들
export const supabase = {
  // 📖 읽기 (Read)
  async getProducts() {
    return await fetchJSON(
      `${SUPABASE_URL}/rest/v1/products?order=name`,
      { headers }
    );
  },

  // ✏️ 생성 (Create)
  async saveOrder(order) {
    return await fetchJSON(
      `${SUPABASE_URL}/rest/v1/orders`,
      { method: 'POST', headers, body: JSON.stringify(order) }
    );
  },

  // 🔄 수정 (Update)
  async updateProduct(id, updates) {
    return await fetchJSON(
      `${SUPABASE_URL}/rest/v1/products?id=eq.${id}`,
      { method: 'PATCH', headers, body: JSON.stringify(updates) }
    );
  },

  // 🗑️ 삭제 (Delete)
  async deleteOrder(id) {
    return await fetch(
      `${SUPABASE_URL}/rest/v1/orders?id=eq.${id}`,
      { method: 'DELETE', headers }
    );
  },
};
Supabase는 내부적으로 PostgreSQL 데이터베이스를 사용합니다. REST API를 통해 HTTP 요청으로 데이터를 주고받으며, URL 뒤에 ?id=eq.5 같은 쿼리를 붙여서 원하는 데이터만 필터링합니다.

7-3. 실시간 구독

누군가 주문을 추가하면, 다른 기기에서도 자동으로 화면이 업데이트됩니다.

// App.jsx — Supabase 실시간 구독
useEffect(() => {
  const channel = supabaseClient
    .channel('orders-realtime')
    .on('postgres_changes',
      { event: '*', schema: 'public', table: 'orders' },
      (payload) => {
        // DB에 변화가 생기면 자동 실행!
        refreshOrders();  // 주문 목록 다시 불러오기
      }
    )
    .subscribe();

  return () => channel.unsubscribe();  // 정리 (cleanup)
}, []);

모바일에서 메뉴(≡) 버튼을 누르면 사이드바가 슬라이드로 나타납니다. 문제는 각 페이지 컴포넌트가 AppLayout의 사이드바 상태에 직접 접근할 수 없다는 것입니다.

// 각 페이지에서: "사이드바 열어/닫아!" 신호 보내기

// AppLayout.jsx에서: 신호 수신하기
useEffect(() => {
  const handler = () => setSidebarOpen(prev => !prev);

}, []);

9-3. z-index 계층 (겹침 순서)

모바일에서 여러 요소가 겹칠 때, 누가 위에 올라오는지 정하는 것이 z-index입니다.

z-50 ┃ 모달/다이얼로그 ← 가장 위 (전체화면 팝업) ┃ z-45 ┃ 모바일 사이드바 ← 메뉴 열면 콘텐츠 위에 나타남 ┃ z-40 ┃ 페이지 sticky 헤더 ← 스크롤해도 상단에 고정 ┃ z-30 ┃ 모바일 하단 네비 ← 하단 메뉴바 ┃ z-0 ┃ 일반 콘텐츠 ← 기본 페이지 내용
z-index가 같으면 나중에 렌더링된 요소가 위에 옵니다. 갤럭시 폴드에서 사이드바가 헤더에 가려지던 버그는 둘 다 z-40이어서 발생했고, 사이드바를 z-45로 올려서 해결했습니다.

10-2. 핵심 알고리즘: 유사도 점수 계산

여러 기준으로 점수를 매기고, 가장 높은 점수의 상품을 선택합니다.

기준점수예시
완전 일치+1000"스덴 밴딩 54-45" = "스덴 밴딩 54-45"
부분 포함+100"스덴 밴딩" ⊂ "스덴 밴딩 54-45"
동의어 매칭+80"스텐 밴딩" → "스덴 밴딩"으로 변환 후 매칭
수치 일치+5054 = 54 (파이프 직경)
수치 ±1+3553 ≈ 54 (오타 허용)
순서 일치+40"스덴", "밴딩", "54" 순서대로 나타남
초성 매칭+20ㅅㄷ → "스덴" 매칭
Levenshtein 유사도~30글자 수정 횟수 기반 유사도
Levenshtein 거리란? 한 문자열을 다른 문자열로 바꾸는 데 필요한 최소 편집 횟수입니다. "밴딩" → "벤딩": 1번 변경 (ㅐ→ㅔ) = 거리 1 → 유사도 높음! "밴딩" → "자바라": 3번 변경 = 거리 3 → 유사도 낮음
